What is the SoA?

The Statement of Applicability (SoA) is where you document which Annex A controls you apply (or exclude) and why, plus the implementation status and evidence references. In practice, it’s a “control decision + proof map” that auditors will look at early.

Minimum fields (startup-friendly)

  • Control ID (Annex A reference)
  • Control name
  • Applicability (Applicable / Not applicable)
  • Justification (why you included/excluded it)
  • Implementation status (Not started / In progress / Implemented)
  • Evidence link (policy, log, ticket, runbook, screenshot)
  • Owner (optional but useful)

Template (copy/paste)

Control ID | Control name | Applicable? | Justification | Status | Evidence | Owner
A.5.1      | Policies for information security | Yes | Required ISMS policy set | Implemented | link-to-policy | Security/CTO
A.5.19     | Information security in supplier relationships | Yes | Critical vendors exist | In progress | vendor-assessment-notes | Ops
A.8.1      | User endpoint devices | Yes | Company-managed laptops | Implemented | MDM config + inventory | IT
A.8.12     | Data leakage prevention | No  | No DLP tooling in scope; mitigated via access controls + encryption | Planned | risk acceptance + compensating controls | Security

Common auditor flags (startups)

  • SoA exists but has no evidence references.
  • “Not applicable” justifications are vague (“not relevant”).
  • Scope creep: controls included for systems that are out of scope.
  • Status says “Implemented” but evidence is missing or inconsistent.

How to keep it minimal

  1. Lock scope first (systems, teams, locations).
  2. Map each applicable control to one primary evidence artifact.
  3. Use compensating controls explicitly (and record the risk decision).
